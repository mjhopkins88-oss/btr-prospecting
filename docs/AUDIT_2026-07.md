## Findings

Ranked by severity. Each entry: evidence, impact, proposed fix, effort. "Outcome" is set after Stage 2.

### F1 — Zero-users auth bypass exposes all internal data
- **Evidence:** `app.py:2686`. Live-tested twice: with `users` table empty, `GET /api/multifamily/leads` and `GET /api/multifamily/campaigns` both returned `200` with **zero auth cookie**; after bootstrapping one user, both correctly `401`'d. Independently reconfirmed via static analysis by a separate audit pass.
- **Impact:** if the production `users` table is ever empty while the app is publicly reachable (botched migration, accidental delete, a window between deploy and first admin signup), every `require_auth`-only route — all lead/campaign/activity/outcome/notification data — is open to any anonymous caller.
- **Fix:** bootstrap mode should expose *only* the initial setup path; every other route 401s regardless of user count.
- **Effort:** Small (one conditional in `_enforce_auth`).
- **Outcome:** **RESOLVED** — see Stage 2 / F1 below.

### F2 — Unmerge has no application-level function or endpoint
- **Evidence:** `multifamily/repository.py:1801` (`mark_lead_merged`) tombstones a merged lead; grepping the whole repo for `unmerge`/`restore`/`reactivate` returns nothing. Live test: manually flipping `merge_status`/`merged_into_id` back via raw SQL fully restores visibility (proves the data model supports it), but no `unmerge_lead()` function or route exists anywhere for an operator to actually call.
- **Impact:** an incorrectly auto-merged pair of leads (two genuinely different companies folded together) currently has no supported recovery path short of a manual database edit.
- **Fix:** add an `unmerge_lead(lead_id)` repository function + an admin-only endpoint.
- **Effort:** Medium (new function, route, UI affordance, tests).
- **Outcome:** **DEFERRED — explicitly out of scope this pass.** Added to the post-launch queue as **HIGH priority, first post-launch pass** (see Pilot Launch Gate below).

### F3 — CSV import has no size/type limit
- **Evidence:** `multifamily/campaigns/csv_import.py` (500-row cap exists and works); `api/routes/multifamily.py:1400-1436` reads the whole upload via `.read()` before the row cap applies, with no `MAX_CONTENT_LENGTH` and no `spam_guard.check_payload_size`-style guard on this route (unlike `POST /leads`, which does check size). No content-type/extension check either.
- **Impact:** an authenticated user (or anyone with a valid session) can upload an arbitrarily large or non-CSV file, which is fully read into memory before any cap applies.
- **Fix:** enforce a max upload size (2 MB) and a content-type/extension check ahead of the read, alongside the existing row cap; return a clean 400 for violations.
- **Effort:** Small.
- **Outcome:** **RESOLVED** — see Stage 2 / F3 below.

### F4 — IP-hash salt defaults to a public, hardcoded string
- **Evidence:** `multifamily/spam_guard.py:47` (`_IP_HASH_SALT = 'btr-multifamily-intake-v1'`), used directly in `hash_ip()`'s `sha256(f'{salt}:{ip}')`.
- **Impact:** if `MULTIFAMILY_IP_HASH_SALT` is never set in the production environment, the salt is public (it's in this open-source repo), so anyone can brute-force all ~4.3B IPv4 addresses against the known default and reverse every stored `submitted_ip_hash`. **UNVERIFIABLE** whether the Railway env var is actually set — operator must check the deploy environment directly.
- **Fix:** at startup, if the salt is unset or still equals the hardcoded default, log a prominent warning server-side and surface a visible (admin-only) "salt not configured" banner on the admin Overview. Never block startup; never expose anything publicly.
- **Effort:** Small.
- **Outcome:** **RESOLVED** — see Stage 2 / F2 below (labeled F2 in Stage 2 per the operator's numbering).

### F5 — Merge gap-fill omits `page_variant`/`campaign_id`
- **Evidence:** `multifamily/matching/merge_engine.py:67` — the "fill gaps only" field tuple includes `offer_type` and every UTM field, but not `page_variant`/`campaign_id`.
- **Impact:** when an incoming submission merges into a survivor that has no `page_variant`/`campaign_id` set yet, those two fields never backfill onto the survivor's row (the attribution *touch* history still records them correctly, but `get_source_performance()`'s `leads_by_page_variant`/`leads_by_campaign_id` — which read the lead row, not touch history — can undercount).
- **Fix:** add `'page_variant', 'campaign_id'` to the gap-fill tuple.
- **Effort:** Trivial.
- **Outcome:** **RESOLVED** — see Stage 2 / F4 below.

### F6 — Day-0 renewal falls into the pre-renewal path, not `post_renewal_active`
- **Evidence:** `multifamily/timing/process_stage_detector.py:112` — branches on `renewal_days < 0`; at `renewal_days == 0` (renewal is today) this is `False`, so the lead stays in the pre-renewal `renewal_window`/rescue path rather than being classified `post_renewal_active`.
- **Impact:** a lead whose renewal is literally today gets rescue-posture messaging instead of the softer just-signed posture — arguably defensible either way, but currently undocumented as a deliberate choice.
- **Fix:** decide explicitly (classify day-0 as rescue, since the renewal hasn't definitively closed yet) and add a code comment + test asserting it.
- **Effort:** Trivial (one-line comment/test) once the decision is made.
- **Outcome:** **Decision recorded: classify day-0 as rescue** (current behavior is correct, just undocumented). **Implementation deferred to post-launch** per explicit scope — no code change this pass; added to the post-launch queue.

### F7 — `signal_timeline` doesn't read the `multifamily_signals` table
- **Evidence:** `api/routes/multifamily.py:87` (`_signal_timeline`) — sources exclusively from `lead.signals`/`lead_json`, never from `multifamily_signals`. Live test: a signal inserted directly into `multifamily_signals` (bypassing `lead_json`) never appears in `_serialize_lead(...)['signal_timeline']`, even after a fresh DB reload.
- **Impact:** harmless *today* because every real write path pairs an `insert_signal` call with an `insert_lead`/`update_lead` that keeps `lead_json` in sync — but there is no code-level guarantee enforcing that pairing. A future write path (e.g., an automated SERP/permit collector inserting signals directly) that doesn't also update `lead_json` would silently produce an incomplete timeline with no error.
- **Fix:** either have `_serialize_lead` reconcile against `get_signals_for_lead(lead.id)` for real leads, or add an explicit invariant test.
- **Effort:** Medium (touches the serialization hot path; needs care not to change scoring/timing inputs).
- **Outcome:** **DEFERRED — explicitly out of scope this pass.** Flagged as a **prerequisite for the SERP phase** on the post-launch queue, since SERP collectors are exactly the future write path most likely to insert signals without a paired `lead_json` update.

### F8 — 11 vs. 10 process stages (spec/doc mismatch only)
- **Evidence:** `multifamily/timing/process_stage_types.py:20-32` lists 11 stages (`inbound_request, renewal_window, acquisition_due_diligence, refinance_or_financing, entitlement_or_permit, construction_loan_closing, construction_start, completion_or_lease_up, post_renewal_active, post_renewal, general_watchlist`). Not a code defect — all 11 independently confirmed reachable with no dead states.
- **Impact:** none functionally; a doc/audit-brief count mismatch that could cause confusion in future stage-coverage audits.
- **Fix:** correct the "10 process stages" reference wherever it appears in audit/spec documentation.
- **Effort:** Trivial.
- **Outcome:** **RESOLVED** — see Stage 2 / F5 below.

## New Stage-1 findings (items 12/13, report only — not fixed this pass)

### N1 — `delete_lead()` doesn't cascade, leaving orphaned rows
- **Evidence:** `multifamily/repository.py:579-582` — `delete_lead()` does only `DELETE FROM multifamily_leads WHERE id = ?`. It does not call any of the `delete_signals_for_lead`, `delete_activities_for_lead`, `delete_attribution_for_lead`, `delete_outcomes_for_lead`, `delete_snapshots_for_lead`, `delete_notifications_for_lead`, or `delete_match_candidates_for_lead` helpers that already exist in the same file (contrast with `delete_campaign()` at line 1227, which correctly cascades to `multifamily_campaign_targets`). Live orphan scan against `prospects.db` found orphaned rows in `multifamily_activities`, `multifamily_lead_snapshots`, `multifamily_notifications`, and `multifamily_sales_intelligence_events`; watching the counts grow live during a full test-suite run (each test script's own cleanup calling `delete_lead()` directly) directly confirmed the leak mechanism. No HTTP route in the live app calls `delete_lead()` today (grepped `api/routes/multifamily.py` — zero matches), so this is currently a test-fixture-only leak, not a live-traffic one — but it would affect any future delete-lead feature.
- **Impact:** low today (no user-facing delete-lead path exists yet); would become a real data-hygiene issue the moment a delete feature ships.
- **Proposed fix (not applied):** add `delete_lead_cascade(lead_id)` calling all seven existing `delete_*_for_lead` helpers plus the lead row itself.
- **Effort:** Small.

### N2 — Three routes have zero test coverage at any layer
- **Evidence:** No test script uses a Flask test client (`grep -l test_client scripts/test_multifamily_*.py` → no matches) — only underlying repository/business-logic functions are unit-tested, never the route/auth/serialization layer itself. Three routes have zero coverage even at the logic layer: `GET /match-candidates`, `POST /match-candidates/<id>/merge`, `POST /match-candidates/<id>/dismiss`.
- **Impact:** the admin fuzzy-match-review workflow (an important data-integrity safety valve) had no automated regression coverage.
- **Note:** Journey D (above) exercised `POST /match-candidates/<id>/merge` live through the real HTTP endpoint this session, partially closing this gap for that one route — `dismiss` and the `GET` listing remain uncovered.
- **Proposed fix (not applied):** add a dedicated `scripts/test_multifamily_match_candidates_routes.py`.
- **Effort:** Small.

### N3 — Non-string JSON body values can produce an ungraceful 500
- **Evidence:** `import_campaign_targets` (`api/routes/multifamily.py:1420-1428`) reads `payload.get('csv')` from a JSON body with no type check; a non-string value (e.g. `{"csv": 123}`) reaches `parse_csv_rows`, which calls `.strip()` on it and throws `AttributeError` — verified directly. The app's global `@app.errorhandler(500)` catches it (no hard crash) but returns a generic 500 instead of a clean 400.
- **Impact:** low (requires a malformed JSON body from an already-authenticated caller), but produces a confusing error message instead of clear validation feedback.
- **Proposed fix (not applied beyond the file-upload path already hardened in F3):** add an `isinstance(x, str)` guard before calling `.strip()`/`parse_csv_rows` on the JSON-body `csv` value, returning 400.
- **Effort:** Trivial.
- **Outcome:** **RESOLVED as a side-effect of F3.** The same route edit that hardened the file-upload path (Stage 2 / F3) added an `isinstance(csv_value, str)` guard on the JSON-body path too, since both live in the same function. Live-verified: `POST .../targets/import` with `{"csv": 12345}` now returns a clean `400 {"errors": ["csv must be a string."]}` instead of a 500.
